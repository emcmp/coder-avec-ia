# Cours 2 - Cartographier un codebase

<div className="coursModuleMeta">
  <span>20–30 min</span>
  <span>GitHub Copilot</span>
  <span>Code existant</span>
</div>

<p className="coursModuleLead">
Avant de demander à l’IA d’intervenir dans un projet, il faut comprendre où regarder. Dans cette rencontre, on apprend à repérer les fichiers, responsabilités et risques importants sans lire tout le codebase.
</p>

## 🗺️ Pourquoi cartographier?

Voici une demande fréquente :

<div className="aiBox aiBoxPrompt aiBoxSmall">
  <div className="aiBoxHeader">
    <span className="aiBoxTitle">Demande envoyée à l’IA</span>
    <span className="aiBoxMeta">Trop large</span>
  </div>
  <div className="aiBoxBody">
    <pre><code>{`Ajoute cette fonctionnalité dans le projet.`}</code></pre>
  </div>
</div>

Le problème : l’IA peut répondre vite, mais pas nécessairement au bon endroit.

Sans cartographie, elle risque de :

- viser un fichier secondaire;
- ignorer l’architecture existante;
- dupliquer une logique déjà présente;
- proposer une solution difficile à réviser.

## 🔍 Ce qu’on cherche

Cartographier un codebase, ce n’est pas tout lire.

C’est repérer :

1. le point d’entrée probable;
2. les fichiers liés;
3. les responsabilités principales;
4. les dépendances importantes;
5. les endroits à traiter avec prudence.

La bonne question n’est pas de comprendre tout le projet. La bonne question est de comprendre assez pour agir sans deviner.

## 🤖 Demander une carte de travail

On peut demander à l’IA de produire une première carte.

<div className="aiBox aiBoxPrompt">
  <div className="aiBoxHeader">
    <span className="aiBoxTitle">Prompt de cartographie</span>
    <span className="aiBoxMeta">Exploration</span>
  </div>
  <div className="aiBoxBody">
    <pre><code>{`Aide-moi à identifier les fichiers probablement concernés par cette fonctionnalité.

Pour chaque fichier, indique :
- son rôle probable;
- pourquoi il pourrait être concerné;
- ce qu’il faudrait vérifier.`}</code></pre>
  </div>
</div>

## ✅ Lire la réponse sans tout croire

Une réponse utile doit rester une hypothèse de travail.

<div className="aiBox aiBoxResponse aiBoxSmall">
  <div className="aiBoxHeader">
    <span className="aiBoxTitle">Réponse attendue</span>
    <span className="aiBoxMeta">À vérifier</span>
  </div>
  <div className="aiBoxBody">
    <p>L’IA devrait proposer quelques zones probables, expliquer pourquoi elles sont pertinentes, puis nommer ce qu’il faut vérifier.</p>
  </div>
</div>

À observer :

- les fichiers proposés;
- les hypothèses de l’IA;
- les zones d’incertitude;
- les fichiers à ouvrir ensuite.

:::tip
Une cartographie n’est pas une preuve. C’est une hypothèse de travail à vérifier.
:::

## 🧪 À essayer

On veut faire ce changement :

<div className="aiBox aiBoxPrompt aiBoxSmall">
  <div className="aiBoxHeader">
    <span className="aiBoxTitle">Changement demandé</span>
    <span className="aiBoxMeta">Exercice</span>
  </div>
  <div className="aiBoxBody">
    <pre><code>{`Ajouter un message d’erreur personnalisé quand le formulaire est invalide.`}</code></pre>
  </div>
</div>

Avant de passer à l’action, produis :

- une liste de zones probablement concernées;
- le rôle supposé de chaque zone;
- une question de clarification;
- un risque si on fonce directement.

## 🔩 Récapitulatif

- Cartographier, ce n’est pas lire tout le projet.
- L’objectif est de réduire les hypothèses.
- L’IA peut aider à cartographier, mais il faut vérifier.
- Une bonne cartographie rend les changements plus petits et plus révisables.

## 🚀 Pour aller plus loin

- **Essentiel** : retrouver la zone probable.
- **Consolidation** : lire les dépendances liées.
- **Défi** : comparer une lecture manuelle avec un outil de cartographie.
